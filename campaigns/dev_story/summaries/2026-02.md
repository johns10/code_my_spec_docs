# February 2026: The Harness Reshapes Itself While Shipping MetricFlow

February was the month CodeMySpec stopped being an orchestrator and became a harness. Across 142 commits, the framework that had been driving MetricFlow's build was torn apart and rebuilt around a different premise: that Claude Code is the agent, not a process spawned by a GenServer. Every change in February happened with a real product on the other side of it -- MetricFlow's OAuth integrations, sync pipelines, and dashboards were being shipped from this same harness, day by day, while the harness itself was getting replaced underneath them.

The first week was a controlled demolition. Between February 1st and 6th, John deleted the old session orchestration system in pieces -- context coding sessions, context testing sessions, component spec sessions, component coding sessions, context design sessions, context testing sessions, integration sessions -- each removal taking another orchestrator, finalize step, and spawn handler with it. The `SessionServer`, `CommandResolver`, `EventHandler`, `InteractionRegistry`, and `SessionsBroadcaster` followed, almost four thousand more lines gone in a single trim. In their place came an HTTP-based local server with a router, working-directory plug, and local-only auth, plus a file-based sessions repository and a `SessionStack` for tracking nested agent calls. The agent tasks themselves moved out of `sessions/` into a top-level bounded context, signaling that they were no longer step modules in a state machine but standalone prompt-and-evaluate units.

The second week filled the new shape with content. `DevelopContext`, `DevelopLiveView`, `LiveViewSpec`, `LiveViewTest`, `LiveViewCode`, `DesignUI`, `RefactorModule`, `ResearchTopic`, `TechnicalStrategy`, `LiveContextSpec`, and `DevelopController` all landed as self-contained tasks following the same pattern: a `command/2` that built a prompt, an `evaluate/2` that checked the result against requirements, and a session directory where the prompt was persisted for context. A shared `ProblemFeedback` module standardized how validation failures were piped back to the agent. The `Validation` context was extracted with its own pipeline running compile, tests, Credo, Sobelow, and spec alignment as ordered stages, replaced the old hand-rolled `Quality` module, and grew cassette-based tests for every pipeline path. A full `Transcripts` system arrived on February 8th -- parser, entry, tool call, file extractor -- giving the harness the ability to read Claude Code's own conversation logs and figure out what the agent had touched, with `TaskIdentifier` recovering which task was active from in-band markers.

The third week was where the requirements graph matured. A scoped requirements system landed on the 27th and 28th with a `Registry` and a fleet of checker modules: `BddSpecExistenceChecker`, `BddSpecPassingChecker`, `ArtifactExistenceChecker`, `ComponentCompleteChecker`, `ComponentLinkedChecker`, `StoryQaChecker`, `ProjectQaChecker`, `AllStoriesCompleteChecker`, `ContextReviewValidityChecker`, `SpecTestAlignmentChecker`. Requirements gained story and project scope, and a `ProjectCoordinator.NextActionable` module learned to walk the component tree recursively, returning the next checker that wasn't yet satisfied. By the 28th this fed into a new graph-driven `Dispatch` module that subsumed the old `Orchestrate` step and let `StartImplementation` evaluate the project, pick the next agent task, and fire it idempotently. The harness was no longer running a script -- it was reading the requirements graph and deciding.

The BDD spex layer was rewritten in parallel. February 1st brought a working `Spex` runner with step, failure, and error structs, and a test adapter that could execute scenarios end-to-end. `WriteBddSpecs` was promoted to a first-class agent task on the 1st and refined repeatedly through the month -- syncing the project before generating specs, ensuring boundary files and `shared_givens.ex` existed, allowing component IDs that hadn't been synced to the database, gaining batch execution with a cached spec checker on the 28th. The 618-line `priv/knowledge/bdd/spex.md` reference shipped on the 22nd, alongside companion knowledge articles for Boundary, conventions, HEEx syntax, LiveView core components, forms, patterns, testing, DaisyUI, Tailwind, and a web-cli guide -- the documentation an agent would consult when generating Phoenix code. The MCP servers were quieter this month but converged on the new model: architecture-server tools for spec CRUD were deleted on the 16th in favor of using the agent tasks directly, story tools gained priority fields, and the validators got tightened.

Two infrastructure pieces landed in single commits worth naming. On February 19th, a bootstrap API appeared -- 419 lines in one controller -- giving an agent the ability to authenticate via OAuth, set up a project, and pull configuration through HTTP, the foundation for `claude /init` flows on a fresh machine. On February 24th, a Web Push notification system arrived complete: service worker, push subscriptions, notification client, hook controller, and a `notify` CLI tool, so Claude Code events from a long-running session could surface as desktop notifications. The same day brought `QaApp` and `QaStory` agent tasks -- browser-based verification with state-machine evaluation and issue filing through a new `QaIssue` document parser, the same QA harness that would run against MetricFlow's twenty-five-plus stories two weeks later.

The month closed with consolidation. The `docs/` submodule was migrated to `.code_my_spec/` on the 28th, status files regenerated, framework knowledge symlinked into project directories, and a 170-line Math Test Project seed plus an 8,250-line MetricFlow-and-Fuelytics seed checked in for reproducible demos. February didn't ship a new feature so much as it changed what kind of system CodeMySpec was. The old harness told the agent what to do step by step. The new one handed the agent a prompt, watched what it edited through transcripts, ran the requirements graph against the result, and fired the next task -- and it did all of that while the project being built on top of it kept shipping.
