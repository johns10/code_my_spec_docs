# November 2025: Sessions, Reviews, and a CLI of Its Own

November was the month the harness grew up the stack. The first week brought Context Components Design Sessions -- a parent session type that spawns child component-design sessions and a review session, with a shared finalize step that aggregates results back into the parent. Days later, Context Design Review Sessions landed with their own execute_review step and a context_review_file_checker requirement, giving the system a way to validate that an AI-written design actually conformed to the rules before any code got generated. Similar Components arrived alongside, with a repository, an MCP tool surface, and a LiveView selector -- the harness could now ground its prompts in concrete examples from the same project rather than reasoning in the abstract.

The session machinery hardened underneath all of this. A first cut of session events landed on November 12: an event_handler with three hundred lines of dispatch logic, a session_events table, and an event_type enum. Within a week that grew into a full broadcaster, a Phoenix session channel for live updates, a vscode_channel for IDE integration, and -- after a small data leak in the component_repository got caught and patched -- a security pass on socket and channel joining. By November 21, auto-session support meant the harness could spawn and chain sessions without a user pressing buttons between each one. Context Coding Sessions and Context Testing Sessions filled out the parent-session matrix, each spawning its own children and rolling up results.

Mid-month a different kind of work began: the project setup wizard. November 16 added a GitHub provider that could create code and docs repos for a project; November 19 turned that into a `project_setup_wizard` context with a github_integration module, a script_generator that emits init scripts, and a 799-line setup_wizard LiveView that walked a user through the whole thing. Cassette fixtures for every GitHub create/update path went in alongside, so the wizard's tests could run without hitting the real API. A broad UI refresh on November 22 reworked layouts, navigation, and forms across nearly every LiveView in the app -- the marketing surface and the admin surface got the same coat of paint.

The last week of November was the CLI. On November 26 a fresh `code_my_spec_cli` umbrella appeared with its own application supervisor, a Ratatouille TUI, a navigation component, and a main screen. The next day OAuth landed for the CLI -- a self-hosted callback web server with selfsigned certs, an oauth_client, a strategy module, and the full login/logout/whoami command suite. November 30 brought the heavy lift: a 5,600-line commit titled "FUCKING Cli works" that introduced client_users, project_sync with a file_watcher_server and change_handler, a Specs context for parsing function and field signatures, a UUID migration for components and projects, and the screens and commands that tied the CLI to the running server. Before the day was over a Ratatouille rewrite had reorganized the screens around a components_browser, an init flow, and a repl, and child-component sync was wired into project_sync. The harness now had a terminal-native client, and it could read a real Elixir project's source and round-trip it to the server's component graph.
