# April 2026: The Harness Ships a Second Product

April was the largest month in the project's history -- 354 commits across the harness itself, the marketing site, the plugin distribution, and the second product the harness was now being asked to build. The first half of the month was the harness preparing to be talked about. The second half was the harness being changed by what it had to ship next.

The month opened with a structural reshape of the requirements layer. On April 1 the project removed rollup nodes, turned `Requirement` into a computed struct rather than a persisted row, and pushed the graph into a sigma.js-rendered LiveView with cross-entity edges. The `requirements_repository` was deleted outright. A day later, container tasks landed with a per-component graph view, and the graph projector started fanning out edges based on prereqs rather than hardcoded layouts. The whole "what does the agent need to do next" calculation moved from a database query to a graph traversal. By April 4, an embeddings pipeline with ONNX serving, knowledge MCP tools, and an external-issues triage workflow with realtime channels were all in. Issues got a LiveView. Architecture got visualization tools. The marketing site got hero images for thirty-plus content pages, an `llms.txt` controller, and a methodology page rebuild.

The middle stretch was infrastructure. April 9 through 15 was given over to making the plugin actually distributable: GitHub-Actions-driven extension releases, auto-update on first SessionStart, hooks that nohup their own restart, a daemon with `cms status / start / stop / logs` subcommands, and a CI matrix that knew how to build a Burrito-packaged binary on macOS M1. The `QaJourneyPlan` / `QaJourneyExecute` / `QaJourneyWallaby` trio was collapsed into one `QaJourney` module. Twenty-four skills were consolidated down to seven with a SkillController router. Social login via GitHub and Google OAuth shipped on April 15, and the `FileWatcherServer` was rewritten to do targeted single-file syncs instead of a full-tree walk on every save. By April 18, `FileSync` was routing all I/O through an `Environments` behaviour, an `InMemoryEnvironment` existed for tests, and the walk was constrained to whitelisted roots. That last change is what made everything else possible: once the harness could be told *which* paths to watch, every per-project knob downstream became cheap.

## Since the r/elixir post (April 19 onward)

The `ProjectConfiguration` table landed on April 19. The same commit that introduced it split the stop hook into a thin `Validation` orchestrator and a separate `TaskEvaluator`, added a memory environment for tests, swapped Earmark for MDEx in markdown parsing, and dropped the first seven Spex files for story 553 -- the harness's own BDD tests, run through cassette-backed pipeline fixtures rather than against a live workspace. By the next morning, the stop-hook response body had been compacted to fit under a 4kB cap with a footer-overflow signal, and the spex pipeline could drive failures end-to-end through recorded compile/exunit/spex outputs. That is the inflection point. From this commit forward, every new feature ships with a folder of `criterion_NNNN_*_spex.exs` files instead of a single rule file.

The bottom half of the week went to plugin hardening (`v1.4.12` through `v1.4.26`): SQLite path resolved at runtime, FileWatcher gated on a Repo readiness probe, the publisher learned to clone-and-merge instead of force-pushing the marketplace branch. Then on April 23 the configuration knobs the schema had been quietly carrying became real -- `spec_validation` and `qa_validation` flipped to per-project modes, the LiveView grew toggles for them, and a Spex Credo deny-list went into `priv/credo_checks/` to keep the BDD suite from rotting. April 24 brought Persona Research as a full feature for story 560: context, MCP tool surface, hosted LiveView, remote sync, the `priv/knowledge/persona_research/` playbook, and twenty-plus criterion-level spex. The same day, Technical Strategy got its own rule-by-rule BDD coverage.

April 25 was the day Market My Spec became a thing. The `/products/market-my-spec` and `/products/code-my-spec` pages shipped with a brand-zone CSS system and shared marketing components. The Three Amigos feature landed in a single 5,500-line commit -- backend, MCP tools, a hosted LiveView for walking story-personas-rules-questions interactively, and twelve criterion-level spex covering the readiness rules. The same Three Amigos pattern was then applied retroactively to issue triage and architecture design over the next twenty-four hours. April 26 was a Credo and Boundary marathon: top-level Boundaries declared on the umbrella roots, fifty-seven warnings ground down to three, the last seven direct `Repo` calls pushed out of `_web` into context modules, and `--warnings-as-errors` turned on in the release CI to lock the boundary in. Real GitHub and Google OAuth response shapes were captured into cassettes, and OAuth signup learned to provision a personal account and self-heal legacy users.

April 27 folded the onboarding wizard into `/app` and dropped the auto-provisioning side-effects from the signup callback, so the magic-link click no longer routed through an intermediate confirmation screen. Mailgun was swapped for Resend. `McpServers` and `Environments` were declared as their own Boundaries. `fix_issues` got its own spex pack for the accepted-QA-issue path. April 28 was the day the harness started using its own BDD tooling on its own code at full strength: `WriteBddSpecs` got a validator, architecture gained a patch mode that splices new components onto an existing graph without redesigning it, and Component Spec / Component Test / Component Code generation each landed with twelve-to-twenty criterion files documenting their evaluator rules. The `similar_components` feature was deleted. So were six dead agent-task modules. Story 561 shipped the actionable-requirements queue that drives the orchestrator. Story 671 -- "Project Configuration Filters the Graph Cleanly" -- made `require_specs`, `require_reviews`, and `require_tests` true per-project toggles that rewire the requirement graph at projection time, with seventeen criterion specs covering the filter, splice, and round-trip cases. The "Agentic Marketing with MCP" blog post went out April 29 with refreshed snapshot stats from real production data. The month closed on April 30 with transitive reduction in the requirement graph, kickoff edges that span a surface's full transitive dep closure, terminal-rule surfacing for `bdd_specs_passing`, and a story-chain serialization fix that confirmed the new graph projector was producing coherent walks for both Code-My-Spec and the Market-My-Spec project being bootstrapped from it.

The two-section split is the point. The first three weeks were the harness becoming legible enough to launch. The last twelve days were the harness being asked to build a second product, discovering that some of its assumptions were Code-My-Spec-shaped, and shipping configuration toggles, the Three Amigos workflow, and a graph projector clean enough to survive both projects feeding it.
