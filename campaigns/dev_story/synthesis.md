# Building CodeMySpec: 1,133 Commits, Eleven Months, and a Harness That Built Itself

## What is CodeMySpec?

CodeMySpec is a requirements-driven development harness, written in Elixir and Phoenix LiveView. It models a software project as a graph of stories, components, and requirements, then guides a coding agent through producing each piece in dependency order: specs first, then tests, then implementations, then design reviews, then a QA pass against real browsers. The agent does the writing. The harness decides what gets written next, evaluates whether the output meets a written specification, and refuses to advance until it does.

What makes CodeMySpec unusual is not the methodology. Plenty of tools promise spec-driven development. What makes it unusual is the test it has been put through: starting in late January 2026, the harness was used to build a separate product, MetricFlow, almost entirely from the inside. Then, starting in mid-April, it was pointed at a second product, Market My Spec, and forced to evolve while shipping it. By May 2026 the framework had built two complete products other than itself, and the architectural changes those products demanded had reshaped the framework into something noticeably different from the version that started the year.

This is the story of that build. It is told from 1,133 commits across eleven months, July 2025 through May 2026. The narrative was synthesized by Claude. The commits were authored by Claude. The specifications and the architectural calls were John's.

## Phase 1: Genesis (July 2025)

CodeMySpec began with a `mix phx.new` on July 1, 2025, and a placeholder `generate_demo` mix task that sat untouched for two days while attention moved to agents. The first agent component file appeared on July 5: an `Agent` struct, an `AgentBehaviour`, and a registry that resolved agent types. That seed -- coding agents as pluggable implementations behind a common interface -- would carry through every redesign that followed. By the time the Claude Code adapter landed four days later it was clear that the rest of the system would grow around the question of what to feed those agents.

Mid-July pivoted to the platform. Phoenix 1.8, `mix phx.gen.auth`, daisyUI, and a 1,935-line Accounts context all landed inside a single week. The pattern was set: contexts arrived fully formed, with their tests, repository, and LiveView shell, and the `Scope` struct quietly grew to carry user, account, and project through the system. Projects and OAuth2 followed. The first MCP endpoint shipped on July 25 -- a stories MCP server with a `create_story` tool and the first integration test against an MCP client. By July 31 the system had stories, components, and the dependencies between them. The graph was starting to form.

## Phase 2: A System That Reasons About Itself (August - September 2025)

August was the month CodeMySpec found its skeleton. A 545-line Architecture LiveView rendered the dependency graph. Tools for architecture health, context statistics, and orphaned-context detection landed in the same window, giving the system its first capacity for introspection. The system could describe a project's architecture, run a checker against each component, and tell you what was done. The skeleton was load-bearing.

It introduced a generic `Orchestrator`, a `StepBehaviour` contract, and the first concrete workflow built on top of it: context design sessions. A `RecordingEnvironment` captured Claude Code interactions to JSON for deterministic replay. The orchestration spine was in place.

## Phase 4: The Harness Ships Its First Product (January 2026)

January was 149 commits and one pivot point. The opening week was restructuring instinct: requirements extracted into their own boundary, dynamic data-driven definitions replacing hardcoded checkers ("FUCK YEAH REQUIREMENTS"), the agent task framework rebuilt around a new SessionServer and an InteractionContext. By January 19 the great unbundling began. Dialyzer was removed, static analysis was refactored into a Problems context, and on January 23 a single commit ripped the entire CLI application out of the main codebase: 63 files deleted, more than 6,400 lines gone. With the CLI gone, the Architecture context became the centerpiece. ## Phase 5: Reshape While Shipping (February 2026)

February was the month CodeMySpec stopped being an orchestrator and became a harness. The premise changed: Claude Code is the agent, not a process spawned by a GenServer. Across 142 commits, the framework that had been driving MetricFlow's build was torn apart and rebuilt around that premise. Every change happened with a real product on the other side of it -- MetricFlow's OAuth integrations, sync pipelines, and dashboards were being shipped from this same harness, day by day, while the harness itself was getting replaced underneath them.

The first week was a controlled demolition. Context coding sessions, context testing sessions, component spec sessions, component coding sessions, the SessionServer, the CommandResolver, the EventHandler, the InteractionRegistry -- each removal taking another orchestrator, finalize step, and spawn handler with it. Almost four thousand lines gone in a single trim. In their place came an HTTP-based local server, a file-based sessions repository, a `SessionStack` for tracking nested agent calls, and agent tasks moved out of `sessions/` into a top-level bounded context.

The second week filled the new shape with content. Eleven new agent tasks landed -- DevelopContext, DevelopLiveView, LiveViewSpec, LiveViewTest, LiveViewCode, DesignUI, RefactorModule, ResearchTopic, TechnicalStrategy, LiveContextSpec, DevelopController -- each a self-contained `command/2` and `evaluate/2` pair. A full Transcripts system arrived on February 8: parser, entry, tool call, file extractor, with a `TaskIdentifier` that recovered which task was active from in-band markers in Claude Code's own conversation logs.

The third week was where the requirements graph matured. A `ProjectCoordinator.NextActionable` module learned to walk the component tree recursively and return the next checker that was not yet satisfied. A graph-driven `Dispatch` module subsumed the old `Orchestrate` step. The harness was no longer running a script -- it was reading the requirements graph and deciding. Two infrastructure pieces landed in single commits: a 419-line Bootstrap API for `claude /init` flows, and a complete Web Push notification system so events from a long-running session could surface as desktop notifications. The same week brought QaApp and QaStory tasks -- the same QA harness that would run against MetricFlow's twenty-five-plus stories two weeks later.

## Phase 6: Eat Your Own Cooking (March 2026)

March was the month CodeMySpec was forced to eat its own cooking. While MetricFlow churned through its QA crunch, every gap in the harness surfaced as a real problem in a real product, and 188 commits worth of fixes, replacements, and rewrites poured back into the framework that had built it. An Issues context gave subagents the ability to file their own reports during runs. A permission-request system let the harness run claude on a remote box and ask the user before a destructive shell command. The BDD specs were converted from LiveViewTest to Wallaby browser testing. By mid-month the full-tree resyncs that had powered earlier validation were replaced with mtime-filtered sync feeding a three-phase validation, and a DirtyTracker on components made repeated runs cheap.

The week of March 21 was the inflection. A new CodeMySpecLocalWeb Phoenix endpoint appeared next to the existing web app -- a LiveView-and-API split running on its own port, served from the user's machine, with a ProjectScope plug, a sync page, and a requirements graph visualization driven by a depends_on edge type and a GraphProjector. Hermes MCP was swapped out for Anubis. Sessions moved from in-memory stacks to a DB-backed task lifecycle. The legacy local_server namespace was deleted -- 2,800 lines of REST controllers replaced by MCP tools running through a single MCP plug. The agent task interface was refactored to `command(scope, task)` instead of `command(scope, session)`, touching every one of the forty-odd agent task modules.

Then the last weekend was the sharpest turn. A new Files schema and a FileSync pipeline landed, validating files inline as the watcher saw them. Then a RequirementCalculator and a RequirementGraph appeared -- computing satisfaction directly from Files plus Problems, with a 20x speedup on graph computation. Once that worked, the old machinery came down: 3,963 lines of legacy sync, validation pipeline, and project_sync deleted in one commit, then 3,029 lines of requirement checkers -- 27 separate checker modules -- deleted in the next. The framework that had spent the previous year shaped around imperative checkers running over a flat requirements list now derived everything from a computed graph over a Files table the watcher kept current. The harness ended March smaller in line count than it started, doing more.

## Phase 7: Two Products at Once (April 1 - April 18, 2026)

April was the largest month in the project's history -- 354 commits across the harness, the marketing site, the plugin distribution, and the second product the harness was now being asked to build. That last change is what made everything else possible: once the harness could be told *which* paths to watch, every per-project knob downstream became cheap.

## The Human-AI Collaboration Model

CodeMySpec was not built by a human who happened to use AI for autocomplete. The vast majority of the 1,133 commits were authored by Claude Code under specifications that John wrote. The architectural decisions, the destructive refactors, the calls to throw out a 27-checker abstraction and replace it with a graph computation -- those were John's. The implementation, the tests, the migrations, and most of this narrative were the agent's.

The methodology made it work. Specifications were written as structured documents with acceptance criteria. Architecture rules were codified in markdown the harness could read into prompt context. The Boundary library enforced dependency rules at compile time. The BDD spex layer provided an executable contract between what the spec said and what the code did -- and starting April 19, that same layer started running against the harness's own modules. By May the framework was running roughly two hundred spex against itself on every change.

The recursive irony of the project is that CodeMySpec was built largely by CodeMySpec. Once the harness could pass agent tasks to Claude Code and evaluate the results, John pointed it at its own backlog. The April 28 Component Spec / Test / Code generators were specified, tested, and partly implemented through the same agent task pipeline they implemented. The harness eats itself.

## Where It Stands

As of May 5, 2026, CodeMySpec is a Phoenix application with two faces -- a hosted web app at codemyspec.com and a local Phoenix endpoint shipped as a Burrito-packaged binary -- exposing its capabilities through both LiveView and MCP. It models projects as a requirement graph derived from a Files projection that the watcher keeps current. It runs around two hundred BDD spex against its own code. It has built MetricFlow end-to-end. It is currently building Market My Spec end-to-end, and the configuration toggles required for that second build are now part of the harness itself.

The trajectory is clear. The pace measured in commits has not slowed -- 354 in April was the highest month yet, and May is on the same shape five days in. But the shape of the work has changed. The destructive rewrites of January through March are mostly done. April and May are about extending the graph projector, the configuration toggles, the agent task surface, and the boundary protections so the framework can host more than one product at once.

What remains is the work of turning a functional harness into a public product. The r/elixir post sits a few days out. After that, the harness has its second case study to point at. The first one already shipped.
