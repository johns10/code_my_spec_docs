# Building CodeMySpec: Eight Months of Human-AI Construction

## Introduction

CodeMySpec is an Elixir/Phoenix platform that helps developers build software from specifications using AI coding agents. It models a project's architecture -- components, dependencies, requirements, behavioral specifications -- and orchestrates Claude Code sessions to design, implement, test, and validate each piece. The system treats AI not as an autocomplete engine but as an autonomous development partner operating within structured constraints: well-crafted prompts in, validated artifacts out.

This is the story of how it was built. Between July 2025 and March 2026, a solo developer and Claude Code produced 584 commits across eight months, iterating from a bare `mix phx.new` scaffold into a platform that can take a user story, decompose it into an architecture, generate behavioral specifications, write code, run quality assurance in a real browser, file issues, triage them, and dispatch fixes -- pausing to ask a human for permission when it needs to do something dangerous.

The journey falls into four distinct phases, each representing a fundamental shift in what the project was and how it thought about the relationship between humans, specifications, and AI agents.

## Phase One: Foundation at Speed (July - August 2025)

The project began on July 1, 2025, and within 29 commits that month, it had a working authentication system, multi-tenant accounts with roles and invitations, a project context, a stories context with full CRUD, an OAuth2 implementation, and an MCP server exposing story management tools to Claude. Every feature arrived fully formed -- context module, repository, schema, migration, LiveView UI, and tests -- in a single commit. The Scope struct, which would become the central nervous system of every request, evolved in lockstep, growing to carry user, account, and project context through the system.

The pace was extraordinary: over 18,000 lines of production code in one month. But what made July remarkable was not the volume -- it was the discipline. The MCP server went through a deliberate refactoring mid-build, converting resources to tools after the developer learned the protocol's strengths in real time. Even at full sprint, architectural choices were being made, not deferred.

August extended the foundation into intelligence. The Components system gave CodeMySpec the ability to model software architecture as a dependency graph, complete with a LiveView visualization and typeahead navigation. A Rules system encoded project conventions as structured, queryable data. The Sessions context established the schema for tracking AI interactions. And the Project Coordinator emerged as the first orchestration layer -- analyzing component readiness by running requirement checkers across the dependency graph, then exposing the results through a JSON API.

These two months established patterns that would survive every subsequent refactoring: the Scope struct as the universal context carrier, bounded contexts as the public API boundary, MCP servers as the interface between the platform and AI agents, and requirement checkers as the mechanism for deciding what to build next.

## Phase Two: The Orchestration Engine (September - October 2025)

September was the month the system learned to think in sequences. A massive commit on September 22nd -- 47 files changed, 1,500 lines added -- introduced the session orchestration pipeline: a generic framework where AI-driven workflows could be defined as typed steps (initialize, generate, validate, finalize), each returning structured results. The pattern proved immediately reusable. Context design sessions and component design sessions were built on the same foundation within days. A recording infrastructure was built alongside it, enabling replay-based testing of AI interactions without calling the AI -- a critical investment in testability that would pay dividends for months.

October was the month the project turned outward. A Dockerfile, Fly.io deployment configs, GitHub Actions CI/CD, and deploy scripts took it off localhost. A home page, marketing layout, and content management system gave it a public face. The content pipeline went through its own rapid evolution, eventually settling on a git-based sync system pulling from a dedicated docs repository. An integrations framework with encrypted credential storage appeared, with GitHub as the first provider and Google Analytics following weeks later.

But the real significance of October was the first complete coding session. On October 1st, the component coding session orchestrator landed -- initialization, test generation, implementation, and test-fixing as orchestrated steps. This was the core loop the entire system existed to support. The architecture that September had built as an abstraction was now generating code.

The commit messages from these months told their own story. The explosive bursts -- eight or nine commits in a single day -- alternated with quieter stretches of one or two. This was the rhythm of a developer working with Claude Code: intense construction sessions where major subsystems materialized, followed by integration and refinement. The AI was not writing boilerplate; it was a collaborator on architectural decisions, generating thousands of lines of tested, structured code in sessions that sometimes ran past midnight.

## Phase Three: The Terminal Breakout and the Great Unbundling (November 2025 - January 2026)

November marked the most dramatic shift in the project's interface philosophy. Over a furious 48-hour stretch, a full terminal application appeared -- built on Ratatouille, with OAuth authentication, project initialization, component browsing, and a REPL screen. The commit message "FUCKING Cli works" captured the developer's energy. Real-time communication arrived in parallel: session events, a broadcaster, authenticated WebSocket channels. By month's end, a developer could sit at a terminal and drive the whole system.

December deepened the terminal story. A tmux adapter gave the CLI the ability to spawn and manage terminal panes programmatically, piping prompts to Claude Code sessions and capturing results. The late-night commit "f7ucking claude works in the tui!!" -- 2,925 lines of new code -- marked the moment the CLI could orchestrate AI coding sessions directly inside tmux. Meanwhile, the old Specs module was absorbed into a unified Documents system, and the database-backed rules system was gutted in favor of a leaner file-based approach. A spec-test alignment quality checker, powered by a purpose-built Elixir AST parser, added the first automated quality assurance capability.

Then came January, the project's most prolific month at 155 commits. Two forces dominated. First, the requirements system was rebuilt from the ground up -- dynamic, data-driven requirement definitions replacing hardcoded rules. Agent tasks crystallized into discrete, composable units: component spec generation, code generation, test generation, full context implementation. Second, on January 23rd, the great unbundling arrived. A single massive commit ripped out the entire CLI application -- 63 files deleted, 6,474 lines removed -- and moved it to a separate repository. The Ratatouille screens, the local OAuth client, the file watcher, every migration: gone.

This was the most consequential architectural decision of the project. Rather than maintaining a bundled terminal client, CodeMySpec became a focused server exposing its capabilities through MCP servers. The CLI had served its purpose -- it proved the orchestration model worked -- but maintaining two applications in one repository was creating friction that slowed both. The unbundling freed the server to evolve its MCP interface as the primary contract with AI agents.

What replaced the CLI's orchestration was simpler and more powerful. An Architecture MCP server arrived the same week -- 22 new files, over 4,000 lines of tools for creating specs, validating dependency graphs, reviewing designs, and managing component metadata. BDD specifications landed as a complete subsystem with parser, projector, scenarios, and steps. By the end of January, the system could take a user story, analyze it architecturally, propose component structures, generate behavioral specifications, and validate the result -- all through MCP tool calls, with no custom terminal UI in the loop.

## Phase Four: The Agent-First Architecture (February - March 2026)

February completed the philosophical transformation that January had started. The old session orchestration system -- the GenServer-driven pipeline that had been the project's spine since September -- was torn out entirely. Over 8,000 lines of orchestrator steps, finalize handlers, and spawn logic were deleted in the first week. What replaced it was radically different: agent tasks as self-contained prompt generators, each with a `command/2` function that assembled a prompt from scope and context and an `evaluate/2` function that validated results against the requirements system.

The shift in metaphor was profound. The old system told the AI what to do step by step, driving it through a predefined pipeline. The new system handed Claude Code a well-crafted prompt, validated its output, and decided what came next. The AI went from being a tool called by the system to being a peer collaborating within constraints. A `ProjectCoordinator` with recursive tree traversal found the next actionable piece of work; a graph-driven dispatch module replaced hardcoded sequences; QA agent tasks added browser-based verification with state machine evaluation and issue filing.

A transcript replay system gave the platform the ability to parse Claude Code conversation logs, extract which files were edited, and detect which agent task was active -- making AI sessions auditable and testable after the fact. The Validation context was extracted as its own bounded context with a pipeline running compilation, tests, Credo, Sobelow, and spec alignment in sequence.

March brought the QA infrastructure to maturity. Issue triage and fix pipelines landed as first-class agent tasks. BDD specs migrated from LiveViewTest to Wallaby browser testing, acknowledging that real UI validation required a real browser. The develop agent tasks were dramatically simplified -- over 4,200 lines deleted -- as the system consolidated around shared implementation patterns.

The most architecturally novel feature arrived on March 5th: a permission request system. A controller, WebSocket, LiveView interface, and Phoenix channel were wired together so that remote Claude Code sessions could pause execution, push a notification to a human, and wait for approval before proceeding. It was the kind of feature that only exists when AI agents are running autonomously against real codebases -- a trust boundary made tangible in code. Issues graduated from document parser artifacts into their own bounded context with schema, repository, and projector, completing the domain model for the quality feedback loop.

## The Human-AI Collaboration Story

Throughout all 584 commits, a consistent pattern emerges in the commit messages and code structure. The explosive construction sessions -- sometimes producing thousands of lines of tested code in a single sitting -- carry the unmistakable signature of a developer working in tight partnership with Claude Code. The raw commit messages ("FUCK YEAH REQUIREMENTS", "hot damn lotta shit works", "FUCKING Cli works") mark the moments of breakthrough, the instant when a complex subsystem clicked into place after an intense collaborative session.

But the collaboration was not one-directional. The project's own architecture was shaped by what the developer learned about working with AI. The shift from a server-driven orchestration pipeline to an agent-first architecture was a direct response to discovering that Claude Code worked better as an autonomous peer than as a step-by-step executor. The MCP server interface, the prompt-and-validate pattern, the permission system -- these were all design decisions informed by the lived experience of building software with an AI partner.

## Where It Stands

As of March 2026, CodeMySpec is a platform that can model a software project's architecture, decompose user stories into component designs, generate behavioral specifications, dispatch AI agents to implement and test each piece, run quality assurance in a real browser, file and triage issues, and request human permission when autonomy meets its limits. The architecture is clean: bounded contexts with explicit boundaries enforced by the Boundary library, MCP servers as the interface layer, a requirements system that gates every transition, and a validation pipeline that holds agents accountable to specifications.

The trajectory is clear. Each phase of the project has moved the AI further from "tool" and closer to "collaborator" -- from MCP tool calls in July, to orchestrated sessions in September, to a terminal UI in November, to autonomous agent tasks with human-in-the-loop approval in March. The system is no longer just generating code from specifications. It is managing the full development lifecycle: design, implementation, testing, quality assurance, issue tracking, and remediation. The 584 commits tell the story of a developer who set out to build a tool and ended up building a development team -- one where the newest member happens to be an AI.
