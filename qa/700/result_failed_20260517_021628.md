## Issues

### Plugin rung outer wrapper missing cms-onboarding class when active or done

#### Severity
MEDIUM

#### Description
When the plugin rung is in `active` or `done` state, its outer element (`[data-test='plugin-rung']`) renders with `class="contents"` rather than `cms-onboarding`. The `cms-onboarding` class is applied to the inner `#cms-onboarding-card` element instead.

The BDD spex for criterion 6055 asserts `[data-test='plugin-rung'].cms-onboarding` — this selector fails in active and done states because the outer wrapper uses CSS `display: contents` to let the inner card's styling show through without an extra box.

The fix is either to: (a) move the `data-test="plugin-rung"` attribute from the outer `class="contents"` wrapper onto the inner `#cms-onboarding-card` element, or (b) add `cms-onboarding` to the outer wrapper alongside `contents` if the visual appearance permits, or (c) update the BDD spec to use a descendant selector (`[data-test='plugin-rung'] .cms-onboarding`). Option (a) is simplest — the `data-test` hook should be on the element that carries the visual contract (the inner card).

Affected states: plugin active (account+project done, no CLI), plugin done (CLI has connected).
Confirmed on both the mid-journey state (fresh user `qa-700-fresh@codemyspec.local` after account+project creation) and the CLI-connected state (`qa@codemyspec.local` with `cli_first_connected_at` set).

Source: `lib/code_my_spec_web/live/app_live/overview.ex`, `plugin_rung/1` function, line 606:
```elixir
<div data-test="plugin-rung" data-state={@state} class="contents">
```
